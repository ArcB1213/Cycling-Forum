# 🚴‍♂️ 职业自行车赛事数据管理与评分系统 - 期末项目总结汇报

## 1. 项目概述

本项目是一个全栈 Web 应用，旨在为自行车运动爱好者提供详尽的赛事数据查询、骑手表现分析以及互动评分功能。系统通过高性能的异步架构，实现了海量历史数据的高效检索与高并发下的用户互动。

## 2. 技术架构

系统采用前后端分离架构，重点针对高并发场景进行了深度优化：

### 前端 (Frontend)

- **框架**: Vue.js 3 (Composition API) + TypeScript
- **构建工具**: Vite
- **核心功能**: 响应式数据展示、动态分页加载、用户认证状态管理、交互式评分组件。

### 后端 (Backend)

- **框架**: **FastAPI** (基于 Python 的高性能异步框架)
- **数据库**: **MySQL 8.0** (存储核心业务数据)
- **ORM**: **SQLAlchemy 2.0** (完全异步化配置)
- **缓存**: **Redis** (用于热点数据加速与高并发缓解)
- **异步驱动**: `aiomysql` + `aioredis`

## 3. 核心功能实现

### 📊 数据管理

- **多维数据模型**: 涵盖赛事 (Race)、届数 (Edition)、赛段 (Stage)、骑手 (Rider) 及成绩 (StageResult) 的复杂关联。
- **高效检索**: 支持按年份、类型、姓名等多条件模糊搜索与分页展示。

### 🔐 认证系统

- **安全存储**: 使用 `passlib` (bcrypt) 进行密码哈希加密。
- **邮箱验证**: 集成异步邮件服务，确保用户注册的真实性。
- **JWT 鉴权**: 基于 Token 的无状态认证机制。

### ⭐ 评分系统 (核心亮点)

- **高性能提交**: 采用 UPSERT (ON DUPLICATE KEY UPDATE) 逻辑，合并查询与更新操作。
- **实时统计**: 维护 `rider_stats` 汇总表，通过数据库原子操作确保统计数据的最终一致性。

## 4. 技术难点与优化方案 (汇报重点)

### 🚀 异步化改造

- **全链路异步**: 从 API 路由到数据库驱动，再到缓存失效逻辑，全部采用 `async/await` 模式，极大提升了系统的 I/O 并发处理能力。
- **事件循环保护**: 针对 Redis 模式匹配删除等潜在阻塞操作，实现了 `invalidate_cache_async`，防止高并发下事件循环被同步操作挂起。

### 🛠️ 数据库性能调优

- **N+1 问题解决**: 深度使用 `selectinload` 预加载技术，将原本数十次的关联查询优化为 1-2 次批量查询。
- **连接池优化**: 针对压测反馈，将数据库连接池调整为 `20 (Size) + 40 (Overflow)`，并优化了超时策略。
- **索引优化**: 针对查询频繁的 `rider_id`, `stage_id`, `rank` 等字段建立了复合索引。

### 📈 压测与稳定性

- **专项压测**: 使用 **Locust** 模拟数百名用户同时进行评分操作。
- **错误追踪**: 开发了结构化的错误记录系统，能够自动捕获并分析高并发下的连接重置 (Status Code 0) 等底层异常。

## 5. 项目成果

- **性能表现**: 经过优化，系统在模拟 500+ 并发用户下仍能保持稳定的响应时间。
- **代码质量**: 实现了完善的异步 API 体系，代码结构清晰，具备良好的可扩展性。
- **文档建设**: 编写了详尽的异步开发指南、压测指南及 API 说明文档。

## 6. 总结与展望

本项目通过对 FastAPI、SQLAlchemy 异步特性的深度实践，成功构建了一个能够应对高并发挑战的数据系统。未来计划引入更复杂的推荐算法，并基于历史成绩进行骑手表现的预测分析。
