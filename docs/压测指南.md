## 🎯 使用 wrk 进行压测

### 基础命令格式

```bash
wrk -t<线程数> -c<并发数> -d<持续时间> <URL>
```

**参数说明**：

- `-t`: 线程数（推荐设置为 CPU 核心数）
- `-c`: 并发连接数（模拟用户数）
- `-d`: 测试持续时间（如 30s, 1m, 2h）
- `--timeout`: 超时时间（默认 1s）
- `--latency`: 显示延迟分布

### 针对本项目的压测命令

#### 1️⃣ 测试同步接口性能

```bash
# 进入 WSL
wsl

# 测试获取赛事列表（同步）
wrk -t4 -c100 -d30s --latency http://localhost:8000/api/races

# 测试获取赛事详情（同步）
wrk -t4 -c100 -d30s --latency http://localhost:8000/api/races/1/editions

# 测试赛段结果（同步，复杂查询）
wrk -t4 -c200 -d30s --latency http://localhost:8000/api/stages/1/results
```

#### 2️⃣ 测试异步接口性能

```bash
# 测试获取赛事列表（异步）
wrk -t4 -c100 -d30s --latency http://localhost:8000/api/async/races

# 测试获取赛事详情（异步）
wrk -t4 -c100 -d30s --latency http://localhost:8000/api/async/races/1/editions

# 测试赛段结果（异步，复杂查询）
wrk -t4 -c200 -d30s --latency http://localhost:8000/api/async/stages/1/results
```

#### 3️⃣ 对比测试脚本

创建 `benchmark.sh`:

```bash
#!/bin/bash

echo "==================== 压测开始 ===================="
echo ""

echo "📊 测试同步接口: GET /api/races"
wrk -t4 -c100 -d30s --latency http://localhost:8000/api/races
echo ""

echo "📊 测试异步接口: GET /api/async/races"
wrk -t4 -c100 -d30s --latency http://localhost:8000/api/async/races
echo ""

echo "📊 测试同步接口（高并发）: GET /api/stages/1/results"
wrk -t4 -c500 -d30s --latency http://localhost:8000/api/stages/1/results
echo ""

echo "📊 测试异步接口（高并发）: GET /api/async/stages/1/results"
wrk -t4 -c500 -d30s --latency http://localhost:8000/api/async/stages/1/results
echo ""

echo "==================== 压测完成 ===================="
```

运行:

```bash
chmod +x benchmark.sh
./benchmark.sh
```

### 关键指标说明

| 指标              | 说明              | 优秀标准   |
| ----------------- | ----------------- | ---------- |
| **Requests/sec**  | 每秒请求数（QPS） | > 1000     |
| **Latency (Avg)** | 平均响应时间      | < 100ms    |
| **Latency (99%)** | 99 分位延迟       | < 200ms    |
| **Stdev**         | 延迟标准差        | 越小越稳定 |

### 性能评级

| 等级    | QPS       | 平均延迟  | 评价           |
| ------- | --------- | --------- | -------------- |
| 🔥 优秀 | > 2000    | < 50ms    | 异步优化良好   |
| ✅ 良好 | 1000-2000 | 50-100ms  | 满足高并发需求 |
| ⚠️ 一般 | 500-1000  | 100-200ms | 需要优化       |
| ❌ 较差 | < 500     | > 200ms   | 存在性能瓶颈   |

## 🐍 使用 Python Locust (可视化)

### 1. 安装

```bash
pip install locust
```

### 2. 创建测试脚本 `locustfile.py`

```python
from locust import HttpUser, task, between

class CyclingAPIUser(HttpUser):
    wait_time = between(1, 3)  # 用户请求间隔 1-3 秒

    @task(3)  # 权重为 3
    def get_races(self):
        """测试获取赛事列表"""
        self.client.get("/api/races")

    @task(2)
    def get_races_async(self):
        """测试异步获取赛事列表"""
        self.client.get("/api/async/races")

    @task(1)
    def get_stage_results(self):
        """测试获取赛段结果"""
        self.client.get("/api/stages/1/results")

    @task(1)
    def get_stage_results_async(self):
        """测试异步获取赛段结果"""
        self.client.get("/api/async/stages/1/results")
```

### 3. 运行 Locust

```bash
# 启动基础测试（Web UI）
locust -f locustfile.py --host=http://localhost:8000

# 启动车手评分专项压测（含详细错误日志）
locust -f locustfile_ratings.py --host=http://localhost:8000
```

### 4. 错误日志与分析 (locustfile_ratings.py 特色)

在运行 `locustfile_ratings.py` 时，系统会自动记录详细的错误信息：

- **日志目录**: `backend/stress_test_logs/`
- **错误日志**: `locust_errors_YYYYMMDD_HHMMSS.log` (包含每个错误的详细堆栈和上下文)
- **错误摘要**: `error_summary_YYYYMMDD_HHMMSS.json` (包含错误分布统计和样本数据)

## ⚙️ 高级压测技巧

### 1. 模拟真实用户行为

```python
# locustfile.py
from locust import HttpUser, task, between, TaskSet

class UserBehavior(TaskSet):
    def on_start(self):
        """用户登录"""
        self.client.post("/api/auth/login", json={
            "email": "test@example.com",
            "password": "password123"
        })

    @task(5)
    def browse_races(self):
        """浏览赛事"""
        races = self.client.get("/api/async/races").json()
        if races:
            race_id = races[0]['race_id']
            self.client.get(f"/api/async/races/{race_id}/editions")

    @task(3)
    def view_rider(self):
        """查看骑手详情"""
        self.client.get("/api/async/riders/1")

    @task(1)
    def search_riders(self):
        """搜索骑手"""
        self.client.get("/api/async/riders?country=France&limit=20")

class WebsiteUser(HttpUser):
    tasks = [UserBehavior]
    wait_time = between(2, 5)
```

## 🎯 预期性能基准

### 当前架构（异步数据库 + Redis 缓存）

| 场景                  | 并发数 | 同步 QPS | 异步 QPS | 提升      |
| --------------------- | ------ | -------- | -------- | --------- |
| 简单查询（缓存命中）  | 100    | 800      | 3000     | **3.75x** |
| 简单查询（无缓存）    | 100    | 200      | 800      | **4x**    |
| 复杂查询（多表 JOIN） | 100    | 80       | 400      | **5x**    |
| 高并发场景            | 500    | 150      | 750      | **5x**    |

### 瓶颈识别

| 指标         | 瓶颈位置   | 优化方案               |
| ------------ | ---------- | ---------------------- |
| CPU > 90%    | 计算密集   | 增加 worker 进程       |
| 内存 > 80%   | 数据量过大 | 添加分页，减少返回数据 |
| 数据库连接满 | 连接池不足 | 增大 `DB_POOL_SIZE`    |
| Redis 延迟高 | 缓存瓶颈   | 使用 Redis Cluster     |
| 网络 I/O 高  | 带宽限制   | 启用 gzip 压缩         |

## 🚨 注意事项

### ⚠️ 不要在生产环境压测

- 可能导致服务不可用
- 应在测试环境或本地进行

### ⚠️ 逐步增加并发

- 从 10 → 50 → 100 → 500 递增
- 观察系统响应和资源使用

### ⚠️ 监控数据库连接

```sql
-- MySQL 查看当前连接数
SHOW STATUS LIKE 'Threads_connected';
SHOW STATUS LIKE 'Max_used_connections';
```

### ⚠️ 检查错误日志

# FastAPI 日志

# MySQL 慢查询日志
