# 🚴‍♂️ 职业自行车赛事数据管理与评分系统 - 期末项目总结汇报

## 1. 项目概述

本项目是一个全栈 Web 应用，旨在为自行车运动爱好者提供详尽的赛事数据查询、骑手表现分析以及互动评分功能。系统通过高性能的异步架构，实现了海量历史数据的高效检索与高并发下的用户互动。

## 2. 技术架构

系统采用前后端分离架构，重点针对高并发场景进行了深度优化：

### 前端 (Frontend)

- **框架**: Vue.js 3 (Composition API) + TypeScript
- **构建工具**: Vite
- **核心功能**: 响应式数据展示、动态分页加载、用户认证状态管理、交互式评分组件。

### 后端 (Backend)

- **框架**: **FastAPI** (基于 Python 的高性能异步框架)
- **数据库**: **MySQL 8.0** (存储核心业务数据)
- **ORM**: **SQLAlchemy 2.0** (完全异步化配置)
- **缓存**: **Redis** (用于热点数据加速与高并发缓解)
- **异步驱动**: `aiomysql` + `aioredis`

## 3. 核心功能实现

### 📊 数据管理

- **多维数据模型**: 涵盖赛事 (Race)、届数 (Edition)、赛段 (Stage)、骑手 (Rider) 及成绩 (StageResult) 的复杂关联。
- **高效检索**: 支持按年份、类型、姓名等多条件模糊搜索与分页展示。

### 🏛️ 论坛系统（新增核心功能）

- **完整发帖功能**：

  - 用户可创建标题+正文的帖子
  - 支持软删除（保留数据完整性）
  - 作者权限控制（仅作者可删除）

- **多级评论架构**：

  - 支持楼层（一级评论）+ 子回复树状结构
  - 递归嵌套评论，无深度限制
  - 实时 WebSocket 推送新评论更新

- **浏览量统计**（Write-Back 策略）：
  - Redis 缓存浏览量，5 分钟批量回写 MySQL
  - 避免行锁竞争，减少数据库压力 **95%**
  - 用户个人中心可查看发表过的帖子

### 🔐 认证系统

- **安全存储**: 使用 `passlib` (bcrypt) 进行密码哈希加密。
- **邮箱验证**: 集成异步邮件服务，确保用户注册的真实性。
- **JWT 鉴权**: 基于 Token 的无状态认证机制。

### ⭐ 评分系统 (核心亮点)

- **高性能提交**: 采用 UPSERT (ON DUPLICATE KEY UPDATE) 逻辑，合并查询与更新操作。
- **实时统计**: 维护 `rider_stats` 汇总表，通过数据库原子操作确保统计数据的最终一致性。

## 4. 技术难点与优化方案 (汇报重点)

### 🚀 论坛评论查询性能优化（优先级 1 已完成）

#### 问题分析

- **原始方案**：递归加载评论，每个一级评论触发一次 SQL 查询
- **规模影响**：50 条评论需要 50 次数据库往返，大量并发下造成严重瓶颈
- **性能指标**：响应时间 > 1s，连接池快速耗尽

#### 优化方案

1. **批量查询优化**：

   - 先查询分页的一级评论 ID（只取 ID，减少数据传输）
   - 再用 `WHERE (comment_id IN (...) OR root_id IN (...))` 一次性查询所有相关评论
   - 在内存中构建树结构（避免递归数据库查询）

2. **数据库索引优化**（5 个新增索引）：

   ```sql
   ✅ idx_forum_posts_deleted_created      -- 帖子列表查询优化
   ✅ idx_forum_comments_query             -- 多条件过滤查询
   ✅ idx_forum_comments_root_created      -- 子回复批量查询
   ✅ idx_forum_comments_parent_created    -- 直接回复查询
   ✅ idx_forum_comments_author            -- 作者评论查询
   ```

3. **浏览量 Write-Back 策略**：
   - Redis 缓存浏览量，每条浏览 `HINCRBY` 原子操作
   - 5 分钟定时任务批量回写 MySQL（`CASE WHEN` 批量更新）
   - 避免行锁竞争，单条评论发表时 comment_count 也只记录在 Redis

#### 性能成果

| 指标           | 优化前   | 优化后     | 提升倍数     |
| -------------- | -------- | ---------- | ------------ |
| 数据库查询次数 | ~50 次   | 1-2 次     | **25-50 倍** |
| 平均响应时间   | ~500ms   | ~4ms       | **125 倍**   |
| 树构建耗时     | 递归阶段 | < 1ms      | **50+倍**    |
| 数据库压力     | 频繁写入 | 5 分钟批量 | **减少 95%** |

### 🔄 异步化改造

- **全链路异步**: 从 API 路由到数据库驱动，再到缓存失效逻辑，全部采用 `async/await` 模式，极大提升了系统的 I/O 并发处理能力。
- **WebSocket 实时推送**：评论创建时异步广播给所有连接的用户
- **事件循环保护**: 针对 Redis 模式匹配删除等潜在阻塞操作，实现了 `invalidate_cache_async`，防止高并发下事件循环被同步操作挂起。

### 🛠️ 数据库性能调优

- **N+1 问题解决**: 深度使用 `selectinload` 预加载技术，将原本数十次的关联查询优化为 1-2 次批量查询。
- **连接池优化**: 针对压测反馈，将数据库连接池调整为 `20 (Size) + 40 (Overflow)`，并优化了超时策略。
- **索引优化**: 针对查询频繁的 `rider_id`, `stage_id`, `rank` 等字段建立了复合索引。
- **软删除优化**：为 `is_deleted` 字段建立复合索引，加快软删除数据的过滤查询

### 📈 压测与稳定性

- **专项压测**: 使用 **Locust** 模拟数百名用户同时进行评分操作和论坛浏览。
- **论坛性能基准**：提供 `test_forum_performance.py` 脚本进行评论查询性能测试
- **错误追踪**: 开发了结构化的错误记录系统，能够自动捕获并分析高并发下的连接重置 (Status Code 0) 等底层异常。
